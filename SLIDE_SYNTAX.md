# jyySlide 幻灯片语法规范（GPT 注入版）

> 用于指导 LLM 生成蒋炎岩 (jyy) 风格的 Web 幻灯片（基于 jyyslide-md / Reveal.js 方言）。
> 你的输出必须是**单一 Markdown 文档**，严格遵守下文所有规则；不要输出任何额外说明。

---

## 0. 输出契约（最重要）

1. 只输出 Markdown 正文，不要 ```markdown 围栏包裹整篇内容。
2. 所有分隔符**必须独占一行**，分隔符前后**必须各有一个空行**（即真实的 `\n<sep>\n`），否则不会被解析。
3. 普通 Markdown 的 `---` 分隔线**不可使用**（它会被识别为水平幻灯片切分符）。如需视觉空行，请用 `<hr>`（在本框架中渲染为"空行"，不是分割线）。
4. 全角空格、emoji 可用；正文支持中英文混排。
5. 解析器是基于字符串 split 的简单方案，**关键字 `---` / `----` / `++++` / `--` / `+++++` 不得出现在正文文字里**。
6. 严禁使用 Setext 风格的 H1（标题下面跟一行 `===` 或 `---`），会被当作分隔符。所有标题一律用 `#` / `##` 形式。

---

```text
- 第二部分

正文……

---

# PART 2
# 第二大节标题

----

## 子页

正文……

---

# 结尾 / Thanks
```

## 2. 分隔符总表（核心）

| 分隔符 | 含义 | 写法（独占一行 + 前后空行） |
|---|---|---|
| `---` （3 个 `-`） | **水平幻灯片** 切分，用于切大章节（PART） | `\n---\n` |
| `----` （4 个 `-`） | **垂直幻灯片** 切分，用于同一章节内的多页 | `\n----\n` |
| `++++` （4 个 `+`） | **渐变 / Auto-Animate 幻灯片** 切分，用于"同一页内容逐步演化"（替换上一页，且元素平滑过渡） | `\n++++\n` |
| `--` （2 个 `-`） | **页内 Fragment**：把当前页切成"依次出现"的多段（用 Space/→ 触发下一段） | `\n--\n` |
| `+++++` （5 个 `+`） | **Front-Matter 分隔符**：仅在文档最开头使用一次，分隔 YAML/JSON 元数据与正文 | `\n+++++\n` |
| `<hr>` | **视觉空行**（不是分割线！）。在一页内用来分段、留白 | `<hr>` |

**结构记忆法**：
- `---` 横着翻（左右切大章）
- `----` 竖着翻（上下切子页）
- `++++` 同一张图渐变（auto-animate）
- `--` 一页内逐条揭示

## 3. Front Matter（封面作者信息）

仅当需要 jyy 风格的"作者 + 多个机构 logo"封面时使用。**必须放在文档最顶部**，并以 `\n+++++\n` 分隔后接正文。

**YAML 写法（推荐）**：

```yaml
author:
  name: Tommy
  url: https://example.com

departments:
  - name: "  机构 A  "
    url: https://a.example.com
    img: ./img/a-logo.png
  - name: 机构 B
    url: https://b.example.com
    img: ./img/b-logo.png
+++++

# 封面标题
```

**JSON 写法**：

```json
{
  "author": { "name": "Tommy", "url": "https://example.com" },
  "departments": [
    { "name": "机构 A", "url": "https://a.example.com", "img": "./img/a.png" }
  ]
}
```

若不需要复杂作者信息，**省略 front matter**，直接在第一页用：

```markdown
# 标题
<hr>

> 副标题或一句话简介

Author：Tommy
```

## 4. 标题与层级约定

- `#` (H1)：用于**水平幻灯片（PART）大节标题**。一页可连用两行 `#` 做"大字标题 + 副大字"，例如：
  ```markdown
  # PART 1 🌶
  # Agentic Coding 基础概念
  ```
- `##` (H2)：用于**垂直幻灯片子页**的标题。
- `###`、`####`：尽量避免，必要时用于子页内层级。
- 标题尽量短（≤ 16 字），避免折行。

## 5. 文本格式

| 效果 | 写法 | 备注 |
|---|---|---|
| 加粗 | `**文本**` | 原生 Markdown |
| 斜体 | `*文本*` | 原生 Markdown |
| 行内代码 | `` `code` `` | 原生 Markdown |
| 删除线 | `~~文本~~` 或 `<del>文本</del>` | pymdownx.tilde |
| 高亮 | `==文本==` 或 `<mark>文本</mark>` | pymdownx.mark（与 CJK 字符直接相邻也生效） |
| 下标 | `H~2~O` | pymdownx.tilde |
| 上标 | `E=mc^2^` | pymdownx.caret |
| 插入 | `^^文本^^` | pymdownx.caret，渲染为 `<ins>` |
| Emoji 短代码 | `:rocket:` `:fire:` `:white_check_mark:` | pymdownx.emoji，渲染为 Unicode 字符 |
| 标红（jyy 风格） | `<red>文本</red>` | jyy 自定义样式 |
| 视觉空行 / 分段 | `<hr>` | **不是分割线**，是空行 |
| 引用 / 旁注 | `> 文本` | 小字灰色，常用于补充说明、引用、tip |
| 嵌套引用 | `>> 文本` | 二级引用 |
| 脚注 | `论断[^1]` + 文档末 `[^1]: 出处` | Python-Markdown footnotes |
| 定义列表 | `术语\n: 释义` | Python-Markdown def_list |

> 例子：
> ```markdown
> 这是 <red>必须重点强调</red> 的内容；这是 <del>过时的写法</del>；这是 <mark>关键术语</mark>。
> ```

## 6. 列表

- 无序列表用 `-` 或 `+`（保持单文档内一致，推荐 `-`）。
- 有序列表用 `1.` `2.` `3.`。
- 嵌套用 **4 个空格** 缩进。
- 列表前必须有空行，否则不会渲染成列表。

```markdown
- 第一条
- 第二条
    - 嵌套子项
    - 嵌套子项
- 第三条

1. 步骤一
2. 步骤二
3. 步骤三
```

**任务列表**（pymdownx.tasklist）：

```markdown
- [x] 已完成
- [ ] 待办
- [ ] 进行中
```

**列表逐条揭示**：见 § 12.4 `:::incremental`，比手写 `--` 干净得多。

## 7. 表格

原生 Markdown 表格，**列宽自动**，适合做对比 / 命令速查 / 概念表。

```markdown
| 概念 | 一句话 |
|---|---|
| Token | 模型处理与计费的最小单位 |
| Context | 模型一次能看见的工作内存 |
```

- 对齐：`|:---|`（左）、`|:---:|`（居中）、`|---:|`（右）。
- 单元格内可写 `**粗体**`、`` `code` ``、`<br>` 换行、emoji。

## 10. 图片

**优先使用 HTML `<img>` 标签**，可控制对齐与大小；原生 `![]()` 也支持但不可控样式。

| 用法 | 写法 |
|---|---|
| 居中（块级、独占一行） | `<img class="center" src="path" width="500px">` |
| 右对齐浮动（文字环绕） | `<img class="float-right" src="path" width="250px">` |
| 自定义宽度 | `width="678px"` 或 `width="50%"` |
| 服务端上传图片占位 | `/media/uploads/xxx.png` |
| 原生写法 | `![alt](path)` 或 `![499](path)`（数字会被当 alt） |

```markdown
<img class="center" src="./img/architecture.png" width="600px">

<img class="float-right" src="./img/me.jpg" width="200px">

正文文字会从右图左侧环绕……
```

> 注意：`class="float-right"` 的图片**不占文本流位置**；`class="center"` 会独占一行并居中。

## 11. 引用 / 链接 / 注释

- 链接：`[文本](https://example.com)` 或裸链 `<https://example.com>`。
- 块引用：行首 `>`，可叠加 `>>`。块引用常用于"提示"、"slogan"、"作者备注"。
- HTML 注释 `<!-- ... -->` 不会被渲染，可用于在源文中写工作笔记。

## 12. 块级指令（`:::` 系列）

幻灯片支持一组 `::: <keyword> [args]` 开 / `:::` 关的块级指令，复用同一套状态机：

| 关键字 | 用途 |
|---|---|
| `columns` | 左右/多列布局 |
| `note` / `tip` / `warning` / `danger` / `success` | 五种提示框 |
| `notes` | Reveal.js 演讲者备注（听众看不到） |
| `incremental` | 列表逐条揭示 |
| `timeline` | 时间线 |

共用规则：

- 必须**在同一张幻灯片内闭合**，不能跨 `---` / `----`。
- 三种行（`::: keyword`、`::: column`、`:::`）都必须**独占一行**。
- 代码栅栏内的 `:::` 字面量原样保留。
- **未知关键字**（如 `::: nope`）原样输出，**不报错**——前向兼容。
- 未闭合时整块降级为普通 Markdown。

### 12.1 分栏布局（`:::columns`）

在**同一张幻灯片内**做左右/多列布局。比 Tailwind `<div class="flex">` 写起来更短，整体保持 Markdown 风格。

**语法**：

```text
::: columns 40/60
::: column
左侧 Markdown（任意内容：列表、代码、公式、图片……）
:::
::: column
右侧 Markdown 或图片
:::
:::
```

- `::: columns RATIO` 开块；`RATIO` 形如 `40/60` / `1/2` / `30/30/40`。
- `::: column` 开列，`:::` 关列；外层 `:::` 关整块。
- 三种指令行都必须**独占一行**（允许行首/尾空白）。

**比例规则**：

- 用 CSS Grid `fr` 单位归一化，**不必加起来等于 100**：`40/60`、`2/3`、`1/2/1` 都行。
- 省略比例（`::: columns` 后不写数字）时按列数均分。
- 列数任意：写几个 `::: column` 就有几列。
- 比例数字不够补 `1`，多余的丢弃：`::: columns 40/60` 配三列 → `40fr 60fr 1fr`。

**列内可以放什么**：任何 Markdown（列表、表格、代码块、行内/行间数学、`<img>`、`> 引用`、链接）。每列内部走完整的渲染管线，可以再嵌套 `:::tip` 等其他 directive。

**隔离与降级**：

- 代码栅栏内的 `::: columns` 字面量原样保留，不会被解析。
- 数学公式 `$...$` / `$$...$$` 里的 `:::` 不会被识别。
- 未闭合（缺少结尾 `:::`）或空块时降级为普通 Markdown，**不报错**。
- ⚠️ 必须**在同一张幻灯片内闭合**，**不能跨 `---` / `----`**——分页早于分栏解析。

**与 H1 的关系**：`# 大字标题` 出现在 column 内时，框架不会再用 `.center.middle`（728px 垂直居中容器）包裹整列。但仍**不推荐**在分栏里写 H1——分栏页通常是"内容页"，标题用 `##` 即可。

**典型用例**：

```markdown
# 左文右图
::: columns 55/45
::: column
### 结论

- 一句话要点
- 两到三条支撑
:::
::: column
<img class="center" src="./img/arch.png" width="400px">
:::
:::

# 三列对照
::: columns 1/1/1
::: column
### 旧做法
:::
::: column
### 新做法
:::
::: column
### 建议
:::
:::
```

**与 Tailwind 两栏对比**：

- `:::columns` 更短、Markdown 风格统一，适合纯内容分栏。
- Tailwind `<div class="flex">` / `<div class="grid grid-cols-2">` 仍可使用，适合复杂自定义样式（背景色、border、自定义 gap 等）。
- 两者**不要混用**在同一块里。

### 12.2 提示框 / Admonitions

```text
::: tip
- 默认标题：技巧 💡
- 内容是任意 Markdown
:::

::: warning 自定义标题
关键字后面可以写自定义标题
:::
```

五种类型 → 配色与默认标题：

| 关键字 | 图标 | 默认标题 | 配色 |
|---|---|---|---|
| `note` | 📝 | 提示 | 蓝 |
| `tip` | 💡 | 技巧 | 绿 |
| `warning` | ⚠️ | 警告 | 橙 |
| `danger` | 🚫 | 危险 | 红 |
| `success` | ✅ | 成功 | 青绿 |

适合放：金句、注意事项、踩坑警告、阶段性结论。比 `> blockquote` 更醒目，但**不要每页都用**，否则失去强调感。

### 12.3 演讲者备注（`:::notes`）

```text
::: notes
- 听众看不到（Reveal.js 默认 CSS 把 .notes 隐藏）
- 但内容仍保留在源文里，方便起草 / 留存数据来源
:::
```

直接映射到 Reveal.js 原生 `<aside class="notes">`。比 `<!-- HTML 注释 -->` 多一层语义：以后只要在模板里挂上 RevealNotes 插件（项目当前未打包），即可获得"按 S 在第二屏看备注"能力。当前的实际效果是**隐藏但保留**。

### 12.4 列表逐条揭示（`:::incremental`）

```text
::: incremental
- 第一条
- 第二条
- 第三条
:::
```

块内的**第一个 `ul` 或 `ol`** 的每个 `li` 自动加 `class="fragment"` + 顺序索引。等价于在每条之间写 `--`，但语法干净得多。如果块内没有 list，不报错、原样输出。

适合：操作步骤、要点逐句展开、"先抛问题再揭示答案"。

### 12.5 时间线（`:::timeline`）

```text
::: timeline
- 2023.06: 项目立项
- 2024.01: **MVP** 上线
- 2024.07: 引入 [Claude Code](https://claude.com/code)
- 14:30: 下午活动
- 2025.11：当前阶段
:::
```

每行 `- DATE: EVENT` 渲染为一条时间线节点。视觉是"日期列 + 圆点 + 事件列"的三栏网格，日期等宽字体右对齐到同一列；最后一项的圆点带更大柔光，暗示"当前"。

**分隔符规则**（重要）：

- ASCII `:` **必须后跟空白**（空格 / tab / 行尾）才算分隔符。
- 全角 `：` 后空白可有可无。
- 这条规则避免吃掉 URL（`https://...`）和时间戳（`14:30: 下午`）里的冒号。

**EVENT 走完整 Markdown**：粗体、链接、行内代码、`==高亮==`、`:emoji:` 都生效。

**降级**：不符合 `- DATE: EVENT` 格式的行整行进 event 字段，date 留空，不报错。

## 13. 渐变页（`++++`）的正确用法

适合"同一张图/流程的渐变演化"——上一页保留共同元素，下一页新增/修改少量元素，Reveal 会做平滑过渡。

```markdown
----

<img class="center" src="./img/step1.png">

<hr>

> 第一步：初始状态

++++

<img class="center" src="./img/step2.png">

<hr>

> 第二步：加入新节点

++++

<img class="center" src="./img/step3.png">

<hr>

> 第三步：连边
```

要点：
- 每个 `++++` 之间都是一个**完整的子页**（和 `----` 切的子页一样可独立排版）。
- `++++` 仅在**当前一组垂直幻灯片**（即两个 `----` 或 `---` 之间）内生效。
- 元素能否平滑过渡取决于 DOM 结构相似度——尽量让相邻渐变页**结构一致、只改少数元素**。

## 14. Fragment（`--`）的正确用法

让一页里的内容"依次出现"。按 Space / → 触发下一段。

```markdown
## 渐进揭示

第一段先出现。

--

第二段（按一下空格才出现）。

--

- 列表也可以
- 整段一起出现

--

> 最后再补一句旁白。
```

要点：
- `--` 切分的是"从该位置到下一个 `--` 或本页结尾"为一个 fragment。
- 不要和 `----`（垂直分页）搞混：**只 2 个减号**才是 fragment。

## 16. 风格与写作约定（jyy 风格）

1. **简短**：每条 bullet 一句话，长句拆成两条；标题 ≤ 16 字。
2. **少废话多事实**：先抛结论，再用 1–3 条要点支撑。
3. **善用 `> blockquote`**：放金句、提示、出处、警告。
4. **善用对照表**：概念辨析、命令速查、A vs B 选型。
5. **代码块要能直接复制运行**，配 1–3 条说明 bullet。
6. **emoji 点缀**：`🌶` `🤖` `🔔` `✅` `📌` 可用于章节标题尾部，每页 ≤ 1 个，避免过度。
7. **每个大章节封面页**只放两行 H1（`# PART N 🌶` + `# 章节名`），不放任何其他内容。
8. **不要在正文里写"如上图所示"** —— 图就在那里，直接说重点。
9. **专有名词中英文混排**保留英文：`Context Window`、`Prompt Caching`、`MCP` 等。
10. 中文与英文/数字之间**不强制加空格**，但要全篇统一。

## 17. 常见错误自查清单（生成前自检）

- [ ] 分隔符（`---` / `----` / `++++` / `--` / `+++++`）是否都独占一行，且前后各有一个空行？
- [ ] 正文里是否意外出现了 3 个以上连续的 `-` 或 `+`？
- [ ] 是否使用了 Setext 标题（标题下跟 `===` / `---`）？ → 必须改用 `#` 形式。
- [ ] 是否每个 `---` 后的第一页都是"H1 大章节"？
- [ ] 是否每个 `----` 后的子页都以 `## 子页标题` 开头？
- [ ] 代码块是否标了语言？
- [ ] 图片是否用 `<img>` 控制了对齐和大小？
- [ ] 表格的表头分隔行是否完整（每列都有 `---`）？
- [ ] 子页内容是否过满（超过一屏要拆页或用 `--` 渐进揭示）？
- [ ] 用到 `:::columns` 时，外层 `:::` 与每个 `::: column ... :::` 是否都已闭合，且全部位于**同一张幻灯片**（未跨 `---` / `----`）？
- [ ] `::: <keyword>` / `::: column` / `:::` 是否都**独占一行**？
- [ ] 用到 `::: tip/note/warning/danger/success/notes/incremental/timeline` 时关键字是否拼对（输错会被当作普通文本输出，不报错但你看不到样式）？
- [ ] `:::incremental` 块里是否真包了一个 list（不然不会有 fragment 效果）？
- [ ] `:::timeline` 每行是否符合 `- DATE: EVENT` 格式？
- [ ] 文档是否最终以一个 Thanks / Q&A 页收尾？

## 19. 不支持 / 已知限制

- ❌ 代码块特定行高亮（` ```js{1,3} ` 之类）。
- ❌ Setext 风格 H1 / H2。
- ❌ 正文里出现 `---` / `----` / `++++` / `--` / `+++++` 字符串作为文本内容。
- ❌ Mermaid / PlantUML / 自定义组件 —— 请改用图片或 ASCII 图。
- ⚠️ `<hr>` 在本框架渲染为"空行"，不是横线，切勿用它做"分割线"语义。
- ⚠️ `++++` 渐变页**只在同一组垂直幻灯片内**生效，跨 `---` 不会有过渡。
- ⚠️ 所有 `:::` 指令（`columns / note / tip / warning / danger / success / notes / incremental / timeline`）必须**在单张幻灯片内闭合**，不能跨 `---` / `----`（分页早于指令解析；跨页会触发降级，整块按普通 Markdown 显示）。
- ⚠️ `~~text~~` 现在解析为删除线（pymdownx.tilde）；之前是字面量。如果旧幻灯片里写过 `~~` 当装饰用，可能需要改写为 `--` 或 HTML 字符。
- ⚠️ `==text==` 现在解析为 `<mark>` 高亮；之前是字面量。
- ❌ 未知的 `:::` 关键字（如 `::: foo`）按原文输出，**不报错**——前向兼容。如果想确认关键字是否生效，看渲染结果里是否出现对应 CSS class。

## 20. 调用提示（给 GPT 的使用建议）

当用户给你一个主题（如"15 分钟讲清楚 React Server Components"），按下列顺序生成：

1. 先列**幻灯片大纲**：封面 → 目录 → 3–5 个 PART → Thanks。
2. 每个 PART 列出 3–6 个子页标题。
3. 按本规范产出完整 Markdown，**只输出 Markdown 正文**，不写解释。
4. 自检第 17 节清单后再交付。
